# HTTP协议完整笔记

## 一、HTTP协议基础知识

### 1.1 概述

HTTP（Hyper Text Transfer Protocol，超文本传输协议）是用于从WWW服务器传输超文本到本地浏览器的传输协议。HTTP是一个应用层协议，由请求和响应构成，是一个标准的客户端/服务器模型。

**主要特点**：
- **支持客户端/服务器模型**
- **简单快速**：客户向服务器请求服务时，只需传送请求方法和路径。请求方法常用的有GET、HEAD、POST。由于HTTP协议简单，使得HTTP服务器的程序规模小，因而通信速度很快
- **无连接**：限制每次连接只处理一个请求。服务器处理完客户端的请求，然后响应，并收到应答之后，就断开连接。这种方式可以节省传输时间
- **无状态**：协议对于事务处理没有记忆能力。如果后续的处理需要用到之前的信息，则必须要重传。为了解决HTTP的无状态特性，出现了Cookie和Session技术

### 1.2 传输类型

HTTP允许传输任意类型的数据对象。正在传输的类型由Content-Type加以标记。常见的Content-Type取值如下：

**常见的媒体格式类型**：
- `text/html`：HTML格式（默认格式）
- `text/plain`：纯文本格式
- `text/xml`：XML格式
- `image/gif`：gif图片格式
- `image/jpeg`：jpg图片格式
- `image/png`：png图片格式

**以application开头的媒体格式类型**：
- `application/xhtml+xml`：XHTML格式
- `application/xml`：XML数据格式
- `application/atom+xml`：Atom XML聚合格式
- `application/json`：JSON数据格式
- `application/pdf`：pdf格式
- `application/msword`：Word文档格式
- `application/octet-stream`：二进制流数据（如常见的文件下载）
- `application/x-www-form-urlencoded`：表单数据被编码为key/value格式发送到服务器（表单默认的提交数据格式）

**上传文件使用的媒体格式**：
- `multipart/form-data`：需要在表单中进行文件上传时使用

### 1.3 URL

HTTP的URL是一种特殊类型的URI，包含了用于定位查找某个网络资源的路径，格式如下：

```
http://host[:port][abs_path]
```

- `http`：表示通过http协议来定位网络资源
- `host`：合法的Internet主机域名或者IP地址
- `port`：指定一个端口，如果缺省，默认是80端口
- `abs_path`：请求资源的URL，如果什么都没写，则浏览器会自动加上`/`作为初始路径

## 二、HTTP请求与响应

### 2.1 HTTP请求

HTTP请求由四部分构成：**请求行、请求头、空行、请求正文**

#### 2.1.1 请求方法

##### GET方法

当客户端要从服务器上读取文档、点开某个链接或通过浏览器输入网址浏览网页时，使用GET方法。

**特点**：
- 请求参数和对应的值附加在URL后面
- 利用一个`?`代表URL的结尾及参数的开始
- 参数用`key=value`键值对的方式书写，参数之间用`&`符号隔开
- 一般GET请求的参数大小受限，最大不超过1024字节
- 参数明文显示在URL上，不适合传递私密数据

##### POST方法

POST方法将请求的参数封装在HTTP请求的请求体中，以名称/值的形式出现，可以传输大量数据。POST请求一般用于表单的数据提交。

**POST请求的四种数据格式**：

1. **application/x-www-form-urlencoded**（最常见）
   - Content-Type被指定为`application/x-www-form-urlencoded`
   - 提交的数据按照`key1=val1&key2=val2`的方式进行编码
   - key和val都进行了URL转码

2. **multipart/form-data**
   - 用于上传文件
   - 使用表单上传文件时，必须让表单的enctype属性值为`multipart/form-data`
   - 既可以上传键值对，也可以上传文件

3. **application/json**
   - 消息主体是序列化的JSON字符串
   - 现代Web API常用格式

4. **text/xml**
   - 使用XML格式传输数据

##### HEAD方法

HEAD就像GET，只不过服务端接收到HEAD请求后只返回响应头，而不会发送响应内容。当需要查看某个页面状态时，使用HEAD非常高效。

#### 2.1.2 GET与POST的区别

| 维度 | GET | POST |
|------|-----|------|
| 参数位置 | 拼接在URL后面 | 放到请求体里面 |
| 长度限制 | 有具体限制，一般不超过1024KB | 理论上没有，但浏览器一般都有界限 |
| 安全性 | 数据明文显示在URL上，安全性较低 | 相对更安全 |
| 数据包 | 产生一个数据包（header和data一并发送） | 产生两个数据包（先发header，服务器响应100 continue，再发data） |

> **注意**：从本质上来说，GET和POST都是TCP连接，并无实质区别。差异主要源于HTTP协议和浏览器的限定。

#### 2.1.3 常见请求头

| 字段 | 说明 | 示例 |
|------|------|------|
| Accept | 可接收的响应内容类型 | `Accept: text/plain` |
| Accept-Charset | 可接收的字符集 | `Accept-Charset: utf-8` |
| Accept-Encoding | 可接受的响应内容编码方式 | `Accept-Encoding: gzip, deflate` |
| Accept-Language | 可接受的响应内容语言列表 | `Accept-Language: en-US` |
| Authorization | HTTP认证信息 | `Authorization: Basic xxx` |
| Cache-Control | 是否使用缓存机制 | `Cache-Control: no-cache` |
| Connection | 优先使用的连接类型 | `Connection: keep-alive` |
| Content-Length | 请求体的长度（8进制） | `Content-Length: 348` |
| Content-Type | 请求体的MIME类型 | `Content-Type: application/x-www-form-urlencoded` |
| Host | 服务器域名和端口号 | `Host: www.example.com:80` |
| User-Agent | 浏览器的身份标识字符串 | `User-Agent: Mozilla/5.0...` |
| Referer | 跳转到当前页面的之前页面 | `Referer: http://example.com` |
| Origin | 跨域资源共享请求 | `Origin: http://www.example.com` |

### 2.2 HTTP响应

HTTP响应由四部分构成：**状态行、响应头、空行、响应体**

#### 2.2.1 状态码

HTTP状态码由三位数字组成，分为五大类：

| 状态码类别 | 含义 | 示例 |
|-----------|------|------|
| 1xx | 指示信息 - 请求已接收，继续处理 | 100 Continue |
| 2xx | 成功 - 请求已被成功接收、理解、接受 | 200 OK |
| 3xx | 重定向 - 要完成请求必须进行更进一步的操作 | 301 Moved Permanently, 304 Not Modified |
| 4xx | 客户端错误 - 请求有语法错误或请求无法实现 | 400 Bad Request, 404 Not Found |
| 5xx | 服务器端错误 - 服务器未能实现合法的请求 | 500 Internal Server Error |

**常见状态码**：
- `200 OK`：客户端请求成功
- `400 Bad Request`：客户端请求有语法错误，不能被服务器理解
- `401 Unauthorized`：请求未经授权
- `403 Forbidden`：服务器收到请求，但是拒绝提供服务
- `404 Not Found`：请求资源不存在
- `500 Internal Server Error`：服务器发生不可预期的错误
- `503 Server Unavailable`：服务器当前不能处理客户端的请求，一段时间后可能恢复正常

## 三、HTTP工作原理

### 3.1 请求/响应步骤

1. **客户端连接到Web服务器**：一个HTTP客户端（通常是浏览器）与Web服务器的HTTP端口（默认80）建立一个TCP套接字连接

2. **发送HTTP请求**：通过TCP套接字，客户端向Web服务器发送一个文本的请求报文，由请求行、请求头部、空行和请求体4个部分构成

3. **服务器接收请求并返回HTTP响应**：Web服务器解析请求，定位请求资源。服务器将资源副本写到TCP套接字，由客户端获取。一个响应由状态行、响应头、空行和响应数据4部分组成

4. **释放TCP连接**：若Connection模式为close，则服务器主动关闭TCP连接，客户端被动关闭连接。若Connection为keep-alive，则该连接会保持一段时间，该时间内可以持续使用该连接

5. **客户端浏览器解析HTML内容**：客户端浏览器首先解析状态行，查看请求是否成功的状态代码。然后解析响应头，告知HTML文档字节数和字符集。最后读取响应数据HTML，根据HTML语法格式化并显示

### 3.2 浏览器输入URL后的完整流程

当在浏览器地址栏键入URL并按下回车后：

1. 浏览器向DNS服务器请求解析URL中的域名所对应的IP地址
2. 解析出IP地址后，根据该IP地址和默认端口80，与服务器建立TCP连接
3. 浏览器发出读取文件的HTTP请求，该请求报文作为TCP三次握手的第三个报文的数据发送给服务器
4. 服务器对浏览器请求作出响应，并把对应的HTML文本发送给浏览器
5. 释放TCP连接
6. 浏览器解析HTML文本并显示内容

## 四、HTTP/1.x的问题

### 4.1 TCP连接数限制

对于同一个域名，浏览器最多只能同时创建6~8个TCP连接（不同浏览器不同）。

**域名分片技术**：将资源放在不同域名下（如二级子域名），以突破连接数限制。

**问题**：
- 每个TCP连接需要DNS查询、三次握手、慢启动
- 占用额外的CPU和内存
- 服务器端容易造成网络拥挤
- 移动端问题更明显

### 4.2 队头阻塞（Head of Line Blocking）

每个TCP连接同时只能处理一个请求-响应，浏览器按FIFO（先进先出）原则处理请求。如果上一个响应没返回，后续请求-响应都会受阻。

**管道化（Pipelining）技术**：尝试解决此问题，但存在诸多缺陷：
- 第一个响应慢还是会阻塞后续响应
- 服务器需要缓存多个响应，占用更多资源
- 浏览器中途断连，服务器可能需要重新处理多个请求
- 必须客户端-代理-服务器都支持

### 4.3 其他问题

- **Header内容多**：每次请求Header变化不大，但缺乏压缩传输优化
- **优化代价高**：为减少请求数需要合并文件、雪碧图、资源内联，但导致单个请求变大、延迟变高
- **明文传输不安全**：数据易被窃取和篡改

## 五、HTTP/2.0：二进制分帧与多路复用

### 5.1 核心特性

HTTP/2.0通过二进制分帧技术实现了真正的多路复用，解决了HTTP层的队头阻塞问题。

#### 5.1.1 二进制分帧层

HTTP/2.0在应用层（HTTP/2）和传输层（TCP）之间增加了一个二进制分帧层：

- 将数据分割成更小的"帧"单元
- 每个请求/响应作为独立的"流"进行传输
- 流拥有自己的编号，帧记录所属流ID（Stream Identifier）

**优势**：
- 实现了真正的多路复用
- 不同传输可以交替进行
- 支持设置传输优先级

#### 5.1.2 多路复用

在一个TCP连接上，可以不断发送帧，每帧的Stream Identifier标明这一帧属于哪个流。接收方根据Stream Identifier拼接每个流的所有帧组成完整数据。

**效果**：
- 单连接上多请求-响应并行
- 解决了队头阻塞问题
- 减少TCP连接数量和慢启动问题
- 同一域名只需创建一个连接

#### 5.1.3 头部压缩（HPACK）

HTTP/2.0使用HPACK算法进行头部压缩：

- 减少重复头部信息的传输
- 使用静态表和动态表存储常用头部
- 压缩率可达80%以上

**适用场景**：
1. 浏览器与服务器支持HTTP/2时自动启用
2. 高频请求的Web应用
3. API接口频繁调用
4. 需要携带大量Cookie或认证头的应用
5. 移动网络或带宽受限环境

#### 5.1.4 服务器推送

HTTP/2.0允许服务器主动推送客户端可能需要的资源：

- 减少客户端请求等待时间
- 推送CSS、JS等关联资源
- 提升页面加载速度

**优势对比资源内联**：
- 客户端可以缓存推送的资源
- 客户端可以拒收推送的资源
- 推送资源可以由不同页面共享
- 服务器可以按照优先级推送资源

#### 5.1.5 其他特性

- **应用层重置连接**：引入RST_STREAM类型的frame，可在不断开连接的前提下取消某个请求的stream
- **请求优先级设置**：每个stream都可以设置依赖和权重，按依赖树分配优先级
- **流量控制**：每个HTTP/2流都拥有自己的流量窗口，限制另一端发送数据

### 5.2 HTTP/2工作原理

#### 5.2.1 帧的结构

所有帧都有一个固定的9字节头部：

```
+-----------------------------------------------+
|                 Length (24)                   |
+---------------+---------------+---------------+
|   Type (8)    |   Flags (8)   |
+-+-------------+---------------+-------------------------------+
|R|                 Stream Identifier (31)                      |
+=+=============================================================+
|                   Frame Payload (0...)                      ...
+---------------------------------------------------------------+
```

**字段说明**：
- **Length**：整个frame的长度，24位无符号整数，默认不超过2^14(16384)字节
- **Type**：定义frame的类型（8 bits）
- **Flags**：帧类型相关的布尔标识（8 bits）
- **R**：保留比特位，发送时必须为0
- **Stream Identifier**：流标识符，31位无符号整数，客户端建立必须为奇数，服务端建立必须为偶数

#### 5.2.2 帧的类型

HTTP/2定义了10种类型的帧：

| 帧类型 | 类型码 | 说明 |
|--------|--------|------|
| DATA | 0x0 | 数据帧，装填主体信息 |
| HEADERS | 0x1 | 报头帧，用来打开流或携带首部块片段 |
| PRIORITY | 0x2 | 优先级帧，指定发送者建议的流优先级 |
| RST_STREAM | 0x3 | 流终止帧，请求取消一个流或表示发生错误 |
| SETTINGS | 0x4 | 设置帧，设置连接参数，作用于整个连接 |
| PUSH_PROMISE | 0x5 | 推送帧，服务端推送 |
| PING | 0x6 | PING帧，判断连接是否可用，测量最小往返时间 |
| GOAWAY | 0x7 | GOAWAY帧，用于发起关闭连接请求或警示错误 |
| WINDOW_UPDATE | 0x8 | 窗口更新帧，执行流量控制 |
| CONTINUATION | 0x9 | 延续帧，继续传送首部块片段序列 |

#### 5.2.3 流（Stream）

流是一个逻辑概念，代表HTTP/2连接中在客户端和服务器之间交换的独立双向帧序列。

**流的特性**：
- 单个HTTP/2连接可以包含多个并发的流
- 两端之间可以交叉发送不同流的帧
- 流可以由客户端或服务器单方面建立和使用
- 流可以由任一方关闭
- 帧在流上发送的顺序非常重要，接收方会把相同Stream Identifier的帧重新组装

**流的状态**：

流具有完整的状态机，包括：`idle`（空闲）、`reserved(local/remote)`（保留）、`open`（打开）、`half-closed(local/remote)`（半关闭）、`closed`（关闭）。

状态转换通过发送或接收特定类型的帧（HEADERS、DATA、RST_STREAM等）触发。

#### 5.2.4 连接前言

在HTTP/2中，要求两端都要发送一个连接前言，作为对所使用协议的最终确认：

- **客户端前言**：内容为`PRI * HTTP/2.0\r\n\r\nSM\r\n\r\n`的序列加上一个可以为空的SETTINGS帧
- **服务端前言**：包含一个可以为空的SETTINGS帧

发送完前言后，双方都向对方发送带有ACK标识的SETTINGS帧表示确认。

### 5.3 HTTP/2优化策略调整

对于HTTP/2来说，以下优化可以弃用：
- 合并文件
- 内联资源
- 雪碧图
- 域名分片

**建议**：使用HTTP/2时尽可能将资源细粒化，文件分解得尽可能散，不用担心请求数多。

## 六、HTTP/3.0：基于QUIC的革命性突破

### 6.1 QUIC协议：HTTP/3的技术基石

HTTP/3.0基于QUIC（Quick UDP Internet Connections）协议构建，彻底解决了TCP层的队头阻塞问题。

QUIC是基于UDP的传输层协议，提供TCP级别的可靠性和有序性，同时消除了TCP的固有缺陷。

### 6.2 核心特性

#### 6.2.1 独立流设计

- 每个请求/响应作为独立的流传输
- 单个流的数据包丢失不影响其他流
- 完美解决TCP层队头阻塞问题

#### 6.2.2 连接迁移能力

- 通过Connection ID标识连接，而非传统TCP的四元组
- 支持网络切换（Wi-Fi→4G→5G）时保持连接状态
- 显著提升移动网络体验

#### 6.2.3 快速连接建立

- 首次连接仅需**1-RTT**完成TLS 1.3加密握手
- 重复访问支持**0-RTT**，复用会话密钥实现零延迟连接

#### 6.2.4 内置TLS 1.3加密

- 所有数据默认加密，消除中间人攻击风险
- 加密与传输握手合并进行，减少连接建立时间
- 前向安全特性保障历史数据安全

#### 6.2.5 灵活的拥塞控制

- 可插拔式拥塞控制算法，支持Cubic、BBR等
- 用户空间实现，允许快速迭代和优化

### 6.3 HTTP/3核心机制

#### 6.3.1 二进制帧与多路复用

延续HTTP/2的二进制分帧设计，在QUIC协议支持下实现真正的多路复用：

- 每个流拥有独立的序列号空间和拥塞窗口
- 丢包仅影响当前流，其他流正常传输
- 支持流优先级调度

#### 6.3.2 QPACK头部压缩

针对QUIC协议优化的头部压缩算法：

- 解决HPACK在多流并行下的阻塞问题
- 每个流独立维护头部压缩上下文
- 减少头部信息传输延迟和带宽占用

#### 6.3.3 前向纠错（FEC）

可选的错误恢复机制：

- 在数据包中添加冗余信息，接收端可恢复少量丢失数据包
- 减少弱网环境下的重传次数
- 提升视频流媒体应用的流畅性

### 6.4 HTTP协议对比

| 特性 | HTTP/1.x | HTTP/2 | HTTP/3 |
|------|----------|--------|--------|
| 传输协议 | TCP | TCP | QUIC（基于UDP） |
| 队头阻塞 | 严重（请求级） | 存在（TCP层） | 完全消除 |
| 连接建立延迟 | 2-3 RTT（TCP+TLS） | 2-3 RTT（TCP+TLS） | 1-RTT首次，0-RTT复用 |
| 加密方式 | 可选（TLS单独握手） | 默认TLS（独立层） | 内置TLS 1.3（传输层集成） |
| 网络迁移 | 不支持 | 不支持 | 支持（Connection ID） |
| 多路复用 | 有限（默认6连接） | 支持（基于TCP流） | 原生支持（独立流） |

### 6.5 HTTP/3应用场景

1. **移动互联网应用**：解决网络切换时的连接中断问题，降低移动网络页面加载延迟，提升App弱网稳定性

2. **视频流媒体与直播**：减少视频卡顿和缓冲时间，支持低延迟直播，提升多用户并发访问体验

3. **金融与在线交易**：保障高频交易场景连接稳定性，减少交易延迟，强化数据传输安全性

4. **物联网与实时交互**：支持低延迟设备通信，提升设备响应速度，强化跨网络设备连接稳定性

5. **跨国业务与远程办公**：优化跨洲际数据传输体验，减少长距离网络传输延迟

---

## 七、协议演进总结

HTTP协议的演进是对网络传输性能瓶颈的持续突破：

1. **HTTP/1.0**：解决基础通信问题，但单连接模型导致性能瓶颈
2. **HTTP/1.1**：引入长连接和管道化，但队头阻塞问题依然存在
3. **HTTP/2.0**：通过二进制分帧和多路复用提升并发性能，但TCP层瓶颈未解决
4. **HTTP/3.0**：基于QUIC协议重构传输层，彻底解决队头阻塞，实现连接速度、安全性和网络适应性的全面升级

随着5G、物联网和实时交互应用的快速发展，HTTP/3将逐渐成为互联网传输的主流协议，为用户带来更快速、稳定和安全的网络体验。

## 附录：HTTP相关概念

### 链接（Connection）
一个传输层的实际环流，建立在两个相互通讯的应用程序之间。

### 消息（Message）
HTTP通讯的基本单位，包括一个结构化的八元组序列并通过连续传输。

### 请求（Request）
一个客户端到服务器的请求消息，包括应用于资源的方法、资源的标识符和协议的版本号。

### 响应（Response）
一个从服务器返回的信息，包括HTTP协议的版本号、请求的状态和文档的MIME类型。

### 资源（Resource）
由URL标识的网络数据对象或服务。

### 实体（Entity）
数据资源或来自服务资源的回映的一种特殊表示方法，可能被包围在一个请求或响应信息中。

### 代理（Proxy）
一个中间程序，可以充当服务器或客户机，为其他客户机建立请求。

### 网关（Gateway）
一个作为其他服务器中间媒介的服务器。

### 通道（Tunnel）
作为两个连接中继的中介程序。

### 缓存（Cache）
响应信息的局域存储。
