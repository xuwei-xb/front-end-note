## 为什么需要性能优化

页面加载时间是任何网站成功的核心因素。研究表明：

- **53%** 的移动用户会放弃加载时间超过 3 秒的网站
- 页面加载每延迟 **1 秒**，转化率下降 **7%**
- 搜索引擎将页面速度作为排名因素（SEO）

**从输入 URL 到页面呈现的完整过程**：

```
用户输入 URL
    ↓
DNS 查询 → TCP 连接（三次握手）→ TLS 握手
    ↓
发送 HTTP 请求 → 接收响应（TTFB）
    ↓
解析 HTML → 构建 DOM 树
    ↓
解析 CSS → 构建 CSSOM 树
    ↓
DOM + CSSOM → 渲染树（Render Tree）
    ↓
布局（Layout）→ 绘制（Paint）→ 合成（Composite）
```

## 性能测试指标

### 核心 Web Vitals 指标

Google 定义的核心 Web Vitals 是衡量用户体验的关键指标：

| 指标 | 全称 | 定义 | 良好 | 需改进 | 差 |
|------|------|------|------|--------|-----|
| **LCP** | Largest Contentful Paint | 页面主要内容块完全可见的时间 | < 2.5s | 2.5s - 4.0s | > 4.0s |
| **FID** | First Input Delay | 用户首次交互到页面响应的时间 | < 100ms | 100ms - 300ms | > 300ms |
| **CLS** | Cumulative Layout Shift | 页面加载过程中布局偏移的累积得分 | < 0.1 | 0.1 - 0.25 | > 0.25 |

> **重要更新**：2024 年起，INP（Interaction to Next Paint）已取代 FID 成为新的交互响应指标。

### 速度指标

| 指标 | 定义 | 良好标准 | 说明 |
|------|------|----------|------|
| **FP** (First Paint) | 浏览器首次向屏幕绘制任何像素的时间 | - | 纯色背景也计入 |
| **FCP** (First Contentful Paint) | 首次绘制文本、图片等内容的时间 | < 1.8s | 用户感知到"有内容了" |
| **LCP** (Largest Contentful Paint) | 视口内最大内容元素渲染完成的时间 | < 2.5s | 替代已弃用的 FMP |
| **TTFB** (Time to First Byte) | 从发起请求到收到响应首字节的时间 | < 800ms | 包含 DNS、TCP、TLS、服务器处理 |

### 运行时性能指标

| 指标 | 定义 | 目标值 |
|------|------|--------|
| **TBT** (Total Blocking Time) | FCP 到 TTI 期间，主线程被阻塞超过 50ms 的总时长 | < 300ms |
| **交互延迟** | 任意用户输入到页面响应的平均时间 | < 50ms |
| **帧率（FPS）** | 页面在滚动、动画等场景下的渲染帧率 | 60fps（16.67ms/帧） |
| **卡顿率** | 长任务（>50ms）阻塞主线程的频率 | < 5 次/分钟 |
| **内存占用** | 页面运行时的内存消耗 | 移动端 < 50MB，桌面端 < 200MB |

### 已弃用的指标

| 指标 | 状态 | 替代方案 |
|------|------|----------|
| **FMP** (First Meaningful Paint) | 2020 年停止支持 | 使用 LCP |
| **TTI** (Time to Interactive) | 已弃用 | 使用 TBT |

## 浏览器原理：性能优化的根基

理解浏览器如何工作，才能做出正确的优化决策。

### 浏览器的组成结构

```
┌─────────────────────────────────────────────────────────────┐
│                      用户界面 (User Interface)               │
│  地址栏、前进/后退按钮、书签、刷新/停止按钮、主页按钮          │
├─────────────────────────────────────────────────────────────┤
│                      浏览器引擎 (Browser Engine)             │
│              在用户界面和渲染引擎之间传送指令                  │
├─────────────────────────────────────────────────────────────┤
│                      渲染引擎 (Rendering Engine)             │
│           负责解析 HTML、CSS，并将内容显示在屏幕上            │
├───────────────┬───────────────┬─────────────────────────────┤
│   网络 (Networking)   │  JavaScript 解释器  │   数据存储      │
│   HTTP 请求等         │  如 V8 引擎         │   Cookie、DB    │
└───────────────┴───────────────┴─────────────────────────────┘
```

### 主流浏览器的渲染引擎

| 浏览器 | 渲染引擎 | JavaScript 引擎 |
|--------|----------|-----------------|
| Chrome | Blink (C++) | V8 (C++) |
| Safari | WebKit (C++) | JavaScriptCore |
| Firefox | Gecko (C++) | SpiderMonkey |
| Edge | Blink (C++) | V8 (C++) |

### 浏览器是多进程的

每打开一个 Tab 页，就相当于创建了一个独立的浏览器进程。

**主要进程**：

1. **Browser 进程**：浏览器主进程，负责界面显示、用户交互、资源管理
2. **渲染进程（Renderer）**：每个 Tab 页一个进程，负责页面渲染、脚本执行、事件处理
3. **GPU 进程**：用于 3D 绘制
4. **插件进程**：每种类型的插件对应一个进程

### 渲染进程是多线程的

渲染进程内部包含多个线程：

#### 1. GUI 渲染线程

- 负责渲染浏览器界面，解析 HTML/CSS，构建 DOM 树和渲染树
- **与 JS 引擎线程互斥**：当 JS 执行时，GUI 线程会被挂起

#### 2. JavaScript 引擎线程

- 负责解析和执行 JavaScript 代码（如 V8 引擎）
- **一个 Tab 页只有一个 JS 线程在运行**
- JS 执行时间过长会阻塞页面渲染

#### 3. 事件触发线程

- 管理事件队列，监听事件，符合条件时将回调函数放入事件队列

#### 4. 定时触发器线程

- `setInterval` 与 `setTimeout` 在此线程中计时
- **注意**：浏览器定时计数器不是由 JS 引擎计数的（JS 是单线程的）
- W3C 规范：`setTimeout` 中低于 4ms 的时间间隔算为 4ms

#### 5. 异步 HTTP 请求线程

- 检测到 XHR 状态变化时，将回调函数放入事件队列

### 渲染线程与 JS 引擎线程互斥

由于 JavaScript 可以操作 DOM，如果在修改元素属性的同时渲染界面，可能导致渲染结果不一致。因此浏览器设置 GUI 渲染线程与 JS 引擎为互斥关系。

**示例**：

```html
<script>
  // 此时 DOM 还未构建，会报错
  document.getElementById("app").style.color = "red";
</script>
<body>
  <div id="app">hello world!</div>
</body>
```

**结论**：JS 如果执行时间过长，会阻塞页面渲染，所以应尽量将 JS 放在 `body` 底部。

### 浏览器的渲染流程

```
HTML 字节流
    ↓
编码转换（字节 → 字符）
    ↓
词法分析（字符 → Token）
    ↓
构建 DOM 树（Token → Node → DOM Tree）
    ↓
同时：解析 CSS → CSSOM 树
    ↓
DOM + CSSOM → 渲染树（Render Tree）
    ↓
布局（Layout/Reflow）：计算节点位置和大小
    ↓
绘制（Paint/Repaint）：绘制节点到图层
    ↓
合成（Composite）：将图层合成为最终图像
```

**关键点**：

- 渲染树只包含**可见节点**（`display: none` 的元素不在渲染树中）
- `visibility: hidden` 的元素**仍在渲染树中**（占据布局空间）

### 重排（Reflow）与重绘（Repaint）

#### 重排（Reflow）

当渲染树中的一部分（或全部）因为元素的**尺寸、布局、隐藏**等改变而需要重新构建。

**触发重排的操作**：

- 调整窗口大小
- 改变字体
- 增加/移除样式表
- 内容变化（如用户输入）
- 操作 class 属性
- 脚本操作 DOM
- 计算 `offsetWidth`、`offsetHeight` 等属性
- 设置 style 属性的值

**触发重排的 CSS 属性**：

```css
/* 盒子模型相关 */
width, height, padding, margin, display, border, min-height

/* 定位与浮动 */
top, bottom, left, right, position, float, clear

/* 文字结构 */
text-align, overflow-y, font-weight, font-family, line-height,
vertical-align, white-space, font-size
```

#### 重绘（Repaint）

当渲染树中的一些元素需要更新属性，但这些属性**只影响外观，不影响布局**。

**只触发重绘的 CSS 属性**：

```css
color, border-style, border-radius, visibility, text-decoration,
background, background-image, background-position, background-size,
outline, box-shadow
```

#### 减少 DOM 操作示例

```javascript
// ❌ 不好：频繁读写交错，触发多次重排
elements.forEach(el => {
  const height = el.offsetHeight; // 读取 → 触发重排
  el.style.height = height + 10 + 'px'; // 写入 → 触发重排
});

// ✅ 好：批量读写分离，只触发 2 次重排
const heights = elements.map(el => el.offsetHeight); // 批量读取
elements.forEach((el, i) => {
  el.style.height = heights[i] + 10 + 'px'; // 批量写入
});
```

**优化建议**：

1. 尽量减少 DOM 操作，使用虚拟 DOM 或 DocumentFragment
2. 批量修改样式，使用 class 或 `cssText`
3. 为动画元素使用 `position: fixed/absolute`，脱离文档流
4. 使用 `transform` 代替 `top/left` 等属性
5. 避免使用 `table` 布局（修改一个元素会影响整个表格）
6. 图片设置明确的宽高，避免加载后触发重排

## 页面加载缓慢的原因分析

### 浏览器层面

#### 网络层面

| 问题 | 说明 | 影响 |
|------|------|------|
| 过多的 HTTP 请求 | 每增加一个元素，页面加载时间增加 25-40ms | 增加连接开销 |
| 资源访问带宽小 | 客户端或服务器端带宽不足 | 下载速度慢 |
| 网页元素过大 | 图片、视频、CSS/JS 文件体积大 | 下载时间长 |
| DNS 解析耗时 | DNS 解析可占请求时间的三分之一 | 延迟首字节时间 |

#### 浏览器渲染层面

| 问题 | 说明 |
|------|------|
| 渲染阻塞 | JS 和 CSS 阻塞 DOM 解析和渲染 |
| 重复渲染 | 频繁的重排和重绘 |
| DOM 结构复杂 | 过深的 DOM 层级影响渲染效率 |

#### 服务端层面

| 问题 | 说明 |
|------|------|
| 硬件配置低 | CPU、内存、磁盘 IO 瓶颈 |
| 服务器软件配置 | 防火墙、内网策略、Nginx 配置不当 |
| 数据库未优化 | 慢查询、缺少索引 |
| 代码效率低 | 循环查询数据库、算法复杂度高 |

### 代码层面

| 问题 | 说明 |
|------|------|
| 未对代码进行打包压缩 | 增加传输体积 |
| 未合并重复请求 | 增加请求数量 |
| JS/CSS 编排位置错误 | 阻塞页面渲染 |
| 未使用异步加载 | 阻塞主线程 |
| 重复、无用代码 | 增加解析和执行时间 |

## 优化策略与具体方案

### 1. 减少 HTTP 请求

**方法**：

- 合并 CSS/JS 文件（注意：HTTP/2 下小文件可不合并）
- 使用 CSS Sprites 合并背景图片
- 内联小图片（Data URL / Base64）

**CSS Sprites 示例**：

```css
.sprite {
  background-image: url('sprite.png');
  background-repeat: no-repeat;
}
.icon-home {
  width: 24px;
  height: 24px;
  background-position: 0 0;
}
.icon-user {
  width: 24px;
  height: 24px;
  background-position: -24px 0;
}
```

### 2. 使用 CDN（内容分发网络）

CDN 将静态资源缓存到全球各地的边缘节点，用户从最近的节点获取资源。

**公共 CDN 资源**：

```html
<!-- Google CDN -->
<script src="https://ajax.googleapis.com/ajax/libs/jquery/3.6.0/jquery.min.js"></script>

<!-- cdnjs -->
<script src="https://cdnjs.cloudflare.com/ajax/libs/jquery/3.6.0/jquery.min.js"></script>

<!-- 国内 CDN -->
<script src="https://cdn.staticfile.org/jquery/3.6.0/jquery.min.js"></script>
```

### 3. 资源压缩

#### Gzip/Brotli 压缩

在 Nginx 中启用 Gzip：

```nginx
gzip on;
gzip_min_length 1k;
gzip_comp_level 6;
gzip_types text/plain text/css application/json application/javascript text/xml;
gzip_vary on;
```

#### 代码压缩

使用 Webpack、Vite 等构建工具自动压缩：

```javascript
// webpack.config.js
module.exports = {
  mode: 'production', // 自动启用压缩
};
```

### 4. 缓存策略

#### 强缓存

| Header | 说明 | 示例 |
|--------|------|------|
| `Cache-Control: max-age=31536000` | 缓存一年 | 带哈希的静态资源 |
| `Cache-Control: immutable` | 资源永不变化 | 进一步防止重新验证 |

#### 协商缓存

| Header | 说明 |
|--------|------|
| `ETag` | 资源的唯一标识符 |
| `Last-Modified` | 资源最后修改时间 |

**配置示例**：

```nginx
# 静态资源强缓存一年
location ~* \.(js|css|png|jpg|jpeg|gif|ico|woff2)$ {
  expires 1y;
  add_header Cache-Control "public, immutable";
}

# HTML 协商缓存
location ~* \.html$ {
  add_header Cache-Control "no-cache";
  etag on;
}
```

### 5. DNS 预解析

在页面加载前预先解析域名，减少 DNS 查询时间：

```html
<meta http-equiv="x-dns-prefetch-control" content="on">
<link rel="dns-prefetch" href="//cdn.example.com">
<link rel="dns-prefetch" href="//api.example.com">
```

### 6. 预连接、预加载、预取

```html
<!-- 预连接：提前建立 TCP 连接和 TLS 握手 -->
<link rel="preconnect" href="https://cdn.example.com">

<!-- 预加载：提前加载关键资源 -->
<link rel="preload" href="critical.css" as="style">
<link rel="preload" href="hero.webp" as="image">
<link rel="preload" href="font.woff2" as="font" type="font/woff2" crossorigin>

<!-- 预取：提前加载下一页可能需要的资源 -->
<link rel="prefetch" href="next-page.js">
```

### 7. CSS 放在头部，JS 放在底部

#### CSS 放在 `<head>` 中

```html
<head>
  <!-- 内联关键 CSS -->
  <style>
    body { margin: 0; }
    .hero { min-height: 100vh; }
  </style>
  
  <!-- 异步加载非关键 CSS -->
  <link rel="preload" href="styles.css" as="style" onload="this.onload=null;this.rel='stylesheet'">
</head>
```

#### JS 使用 defer/async

```html
<!-- defer：HTML 解析完成后，按顺序执行 -->
<script defer src="main.js"></script>

<!-- async：下载完成后立即执行，不保证顺序 -->
<script async src="analytics.js"></script>
```

| 属性 | 下载时机 | 执行时机 | 适用场景 |
|------|----------|----------|----------|
| 无 | 阻塞 HTML 解析 | 下载后立即执行 | 依赖 DOM 的脚本 |
| `defer` | 不阻塞解析 | HTML 解析完成后按顺序执行 | 依赖 DOM 的脚本 |
| `async` | 不阻塞解析 | 下载完成后立即执行（乱序） | 独立脚本（如统计） |

### 8. 图片优化

#### 使用现代格式

```html
<picture>
  <source srcset="image.avif" type="image/avif">
  <source srcset="image.webp" type="image/webp">
  <img src="image.jpg" alt="Description">
</picture>
```

#### 响应式图片

```html
<img
  src="image-800w.jpg"
  srcset="image-400w.jpg 400w, image-800w.jpg 800w, image-1200w.jpg 1200w"
  sizes="(max-width: 600px) 400px, (max-width: 1000px) 800px, 1200px"
  alt="Description"
>
```

#### 懒加载

```html
<!-- 首屏图片立即加载 -->
<img src="hero.jpg" loading="eager" decoding="async">

<!-- 非首屏图片懒加载 -->
<img src="content.jpg" loading="lazy" decoding="async">
```

### 9. 字体优化

```css
@font-face {
  font-family: 'Custom Font';
  src: url('font.woff2') format('woff2');
  font-display: swap; /* 关键：避免 FOIT */
}
```

```html
<!-- 预加载关键字体 -->
<link rel="preload" href="font.woff2" as="font" type="font/woff2" crossorigin>
```

### 10. 避免空 src 和 href

空的 `src` 和 `href` 会触发不必要的 HTTP 请求：

```html
<!-- ❌ 不好 -->
<img src="">
<a href=""></a>

<!-- ✅ 好 -->
<img src="placeholder.jpg">
<a href="javascript:void(0);">Link</a>
<!-- 或使用事件处理 -->
<a href="#" onclick="handleClick(event); return false;">Link</a>
```

### 11. 避免 3xx/4xx 错误

#### 避免重定向

- URL 末尾应该添加 `/` 但未添加，会触发 301 重定向
- 使用完整的 URL 地址

```html
<!-- ✅ 好 -->
<a href="https://www.example.com/products/">Products</a>

<!-- ❌ 不好：会触发重定向 -->
<a href="https://example.com/products">Products</a>
```

#### 避免 404 错误

- 提供 favicon.ico
- 检查所有资源链接是否有效
- 使用路由技术处理不存在的页面

### 12. AJAX 优化

#### 使用 GET 请求

- GET 请求可以被缓存
- GET 只需要一个 TCP 包（POST 需要两个）

```javascript
// ✅ 好：使用 GET
fetch('/api/data?' + new URLSearchParams(params));

// 缓存控制
fetch('/api/data', {
  headers: { 'Cache-Control': 'no-cache' }
});
```

### 13. Cookie 优化

- 减少 Cookie 大小
- 静态资源使用无 Cookie 域名

```
主域名：www.example.com（带 Cookie）
```

## 代码层面优化

### 减少 DOM 操作

```javascript
// ❌ 不好：直接操作 DOM
for (let i = 0; i < 1000; i++) {
  document.body.appendChild(document.createElement('div'));
}

// ✅ 好：使用 DocumentFragment
const fragment = document.createDocumentFragment();
for (let i = 0; i < 1000; i++) {
  fragment.appendChild(document.createElement('div'));
}
document.body.appendChild(fragment);
```

### 使用高效的事件处理

```javascript
// ❌ 不好：为每个元素绑定事件
document.querySelectorAll('.item').forEach(item => {
  item.addEventListener('click', handleClick);
});

// ✅ 好：事件委托
document.querySelector('.list').addEventListener('click', (e) => {
  if (e.target.classList.contains('item')) {
    handleClick(e);
  }
});
```

### 避免强制同步布局

```javascript
// ❌ 不好：读取布局属性后立即修改，导致多次重排
elements.forEach(el => {
  el.style.width = el.offsetWidth + 10 + 'px';
});

// ✅ 好：先批量读取，再批量写入
const widths = elements.map(el => el.offsetWidth);
elements.forEach((el, i) => {
  el.style.width = widths[i] + 10 + 'px';
});
```

### 使用 Web Worker 处理复杂计算

```javascript
// main.js
const worker = new Worker('worker.js');
worker.postMessage({ data: largeDataSet });
worker.onmessage = (e) => {
  console.log('Result:', e.data);
};

// worker.js
self.onmessage = (e) => {
  const result = heavyComputation(e.data);
  self.postMessage(result);
};
```

## 常见误区

| ❌ 错误观点 | ✅ 正确理解 |
|------------|------------|
| "TTI 是最重要的指标" | TTI 已弃用，应关注 LCP、FID、CLS |
| "HTTP 请求越少越好" | HTTP/2 支持多路复用，小文件可不合并 |
| "所有资源都需要缓存" | HTML 应设置较短缓存或使用协商缓存 |
| "图片越清晰越好" | 应根据设备分辨率提供合适尺寸 |
