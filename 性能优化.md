## 性能测试指标

### 核心 Web Vitals 指标

| 指标 | 全称 | 定义 | 良好标准 | 需改进 |
|------|------|------|----------|--------|
| **LCP** | Largest Contentful Paint | 页面主要内容块完全可见的时间 | < 2.5s | 2.5s - 4.0s |
| **FID** | First Input Delay | 用户首次交互到页面响应的时间 | < 100ms | 100ms - 300ms |
| **CLS** | Cumulative Layout Shift | 页面加载过程中布局偏移的累积得分 | < 0.1 | 0.1 - 0.25 |

### 速度指标（Speed Metrics）

#### 首次绘制（First Paint，FP）
浏览器首次向屏幕绘制任何像素的时间点，纯色背景也计入。

#### 首次内容绘制（First Contentful Paint，FCP）
浏览器首次绘制文本、图片（带尺寸）、SVG、非白色 canvas 等内容的时间点。

#### 最大内容绘制（Largest Contentful Paint，LCP）
视口内最大内容元素渲染完成的时间点。LCP 元素通常是：
- 图片
- 视频
- 包含大块文本的块级元素
- 带背景图的元素（<10000px² 的背景图不计入）

#### 互动时间（Time to Interactive，TTI）
页面完全可交互的时间点（主线程空闲超过 50ms 连续 5 秒）。

**注意**：TTI 已被弃用，建议使用 TBT（Total Blocking Time）作为替代指标。

#### 首字节时间（Time to First Byte，TTFB）
从发起请求到收到响应首字节的时间，包含 DNS 查询、TCP 连接、TLS 握手和服务器处理时间。

- 良好：TTFB < 800ms
- 需改进：800ms - 1800ms

#### 总阻塞时间（Total Blocking Time，TBT）
FCP 到 TTI 期间，主线程被阻塞超过 50ms 的总时长。

- 良好：TBT < 300ms
- 需改进：300ms - 600ms

### 网络与资源指标

| 指标 | 说明 | 优化目标 |
|------|------|----------|
| **请求次数** | 页面加载发起的 HTTP 请求数量 | 越少越好（考虑 HTTP/2 多路复用） |
| **请求大小** | 所有资源传输的总大小 | 首屏 < 1MB（压缩后） |
| **资源缓存命中率** | 从缓存加载的资源占比 | > 80% |
| **资源压缩率** | 使用 Gzip/Brotli 压缩后的体积减少比例 | 文本资源 > 70% |

### 运行时性能指标

| 指标 | 定义 | 目标值 |
|------|------|--------|
| **首次输入延迟（FID）** | 用户首次交互到事件回调开始执行的时间差 | < 100ms |
| **交互延迟** | 任意用户输入到页面响应的平均时间 | < 50ms |
| **帧率（FPS）** | 页面在滚动、动画等场景下的渲染帧率 | 60fps（16.67ms/帧） |
| **卡顿率** | 长任务（>50ms）阻塞主线程的频率 | < 5 次/分钟 |
| **内存占用** | 页面运行时的内存消耗 | 移动端 < 50MB，桌面端 < 200MB |
| **CPU 占用** | 页面运行时的 CPU 使用率 | < 30%（空闲时接近 0%） |

### 稳定性指标

| 指标 | 定义 | 目标值 |
|------|------|--------|
| **崩溃率** | 页面或浏览器崩溃的频率 | < 0.1% |
| **错误率** | JavaScript 运行时错误的发生频率 | < 0.5% |

## 浏览器渲染原理

### 关键渲染路径（Critical Rendering Path）

```
用户输入 URL
    ↓
DNS 查询 → TCP 连接 → TLS 握手
    ↓
请求 HTML → 接收响应（TTFB）
    ↓
解析 HTML → 构建 DOM 树
    ↓
解析 CSS → 构建 CSSOM 树
    ↓
DOM + CSSOM → 渲染树（Render Tree）
    ↓
布局（Layout）→ 绘制（Paint）→ 合成（Composite）
```

### 关键阻塞点

1. **HTML 解析阻塞**：同步的 `<script>` 标签会阻塞 HTML 解析

2. **CSS 解析阻塞**：`<link rel="stylesheet">` 会阻塞渲染（但不阻塞 HTML 解析）

3. **JavaScript 执行阻塞**：脚本执行期间主线程被完全占用

4. **字体加载阻塞**：默认情况下，字体加载期间文本不可见（FOIT）

### 优化核心思想

减少关键渲染路径上的阻塞点，尽早渲染可见内容。

## 优化策略

### 1. 减少 render-blocking 资源

| 资源类型 | 优化方法 | 代码示例 |
|----------|----------|----------|
| **CSS** | 内联关键 CSS，异步加载非关键 CSS | `<link rel="preload" href="style.css" as="style" onload="this.onload=null;this.rel='stylesheet'">` |
| **JavaScript** | 使用 `defer` 或 `async`，避免同步脚本 | `<script defer src="main.js"></script>` |
| **字体** | 使用 `font-display: swap` | `@font-face { font-display: swap; }` |

### 2. 资源加载优化

| 优化手段 | 适用场景 | 实现方式 |
|----------|----------|----------|
| **预加载（Preload）** | 首屏关键资源 | `<link rel="preload" href="critical.css" as="style">` |
| **预连接（Preconnect）** | 跨域资源提前建立连接 | `<link rel="preconnect" href="https://cdn.example.com">` |
| **预取（Prefetch）** | 用户可能下一步访问的资源 | `<link rel="prefetch" href="next-page.js">` |
| **懒加载（Lazy Loading）** | 非首屏图片、组件 | `<img loading="lazy" src="image.jpg">` |
| **代码拆分（Code Splitting）** | 减少初始 JS 体积 | `import('./module').then(module => {...})` |

### 3. 字体优化策略

1. **使用 `font-display: swap`**：避免 FOIT（Flash of Invisible Text），允许立即使用回退字体

2. **字体子集化**：只提取实际使用的汉字

3. **字体预加载**：
   ```html
   <link rel="preload" href="font.woff2" as="font" type="font/woff2" crossorigin>
   ```

4. **避免 FOUT（Flash of Unstyled Text）**：设置合适的字体栈：
   ```css
   body {
     font-family: 'Custom Font', -apple-system, BlinkMacSystemFont, 'Segoe UI', sans-serif;
   }
   ```

### 4. 图片与媒体优化

| 优化手段 | 效果 | 实现方式 |
|----------|------|----------|
| **现代格式** | WebP/AVIF 比 JPEG 体积小 30-50% | `<picture><source srcset="image.webp" type="image/webp"><img src="image.jpg"></picture>` |
| **响应式图片** | 根据设备分辨率加载合适尺寸 | `<img srcset="image-400w.jpg 400w, image-800w.jpg 800w" sizes="(max-width: 600px) 400px, 800px">` |
| **懒加载** | 减少首屏加载量 | `<img loading="lazy" src="image.jpg">` |
| **压缩** | 减少文件体积 | 使用 `imagemin`、`sharp` 等工具 |

### 5. 缓存策略

| 缓存类型 | 适用资源 | 缓存时间建议 |
|----------|----------|--------------|
| **强缓存** | 带哈希的静态资源（JS、CSS、图片） | 1 年（Cache-Control: max-age=31536000, immutable） |
| **协商缓存** | HTML 文件 | 短时间（如 1 小时）或 ETag |
| **Service Worker** | 离线访问、预缓存关键资源 | 使用 Workbox |

### 6. 代码层面优化

#### 减少重排（Reflow）

- 避免频繁读写 DOM
- 批量操作样式
- 使用 `transform` 和 `opacity` 进行动画

```javascript
// ❌ 不好
for (let i = 0; i < 100; i++) {
  element.style.left = i + 'px';
}

// ✅ 好
element.style.transform = `translateX(${100}px)`;
```

#### 减少重绘（Paint）

- 使用 `transform` 和 `opacity` 进行动画（GPU 加速）

#### 避免 Layout Thrashing

```javascript
// ❌ Layout Thrashing
elements.forEach(el => {
  const height = el.offsetHeight; // // 读取 ← 触发布局计算
  el.style.height = height + 10 + 'px'; // 写入 ← 触发布局计算
});
// 所以会触发N次布局计算
// ✅ 批量读写
const heights = elements.map(el => el.offsetHeight); // 全部读取 触发 1 次布局计算
elements.forEach((el, i) => {
  el.style.height = heights[i] + 10 + 'px'; // 全部写入 触发 1 次布局计算
});
// 只会触发 2 次布局计算
```

## 具体优化方案

### 1. 关键 CSS 内联

```html
<!DOCTYPE html>
<html>
<head>
  <style>
    /* 只包含首屏渲染必需的 CSS */
    body { margin: 0; font-family: system-ui; }
    .hero { min-height: 100vh; background: #000; color: #fff; }
  </style>
  
  <!-- 非关键 CSS 异步加载 -->
  <link rel="preload" href="styles.css" as="style" onload="this.onload=null;this.rel='stylesheet'">
  <noscript><link rel="stylesheet" href="styles.css"></noscript>
</head>
<body>
  <!-- 页面内容 -->
</body>
</html>
```

### 2. JavaScript 代码拆分

```javascript
// 路由级别的代码拆分（React）
const Home = lazy(() => import('./Home'));
const About = lazy(() => import('./About'));

function App() {
  return (
    <Suspense fallback={<Loading />}>
      <Routes>
        <Route path="/" element={<Home />} />
        <Route path="/about" element={<About />} />
      </Routes>
    </Suspense>
  );
}

// 动态导入非首屏功能
document.getElementById('load-chart').addEventListener('click', async () => {
  const chartModule = await import('./chart');
  chartModule.renderChart();
});
```

### 3. 图片懒加载与响应式

```html
<picture>
  <source srcset="hero.avif" type="image/avif">
  <source srcset="hero.webp" type="image/webp">
  <img 
    src="hero.jpg" 
    srcset="hero-800w.jpg 800w, hero-1200w.jpg 1200w, hero-1600w.jpg 1600w"
    sizes="(max-width: 768px) 100vw, (max-width: 1200px) 80vw, 1200px"
    loading="eager"
    alt="Hero image"
    width="1600"
    height="900"
  >
</picture>

<!-- 非首屏图片懒加载 -->
<img 
  src="content.jpg" 
  loading="lazy" 
  alt="Content image"
  decoding="async"
>
```

### 4. 字体优化

```css
@font-face {
  font-family: 'Noto Sans SC';
  src: url('noto-sans-sc-subset.woff2') format('woff2');
  font-weight: 400;
  font-display: swap; /* 关键优化 */
}

body {
  font-family: 'Noto Sans SC', system-ui, -apple-system, sans-serif;
}
```

```html
<!-- 预加载关键字体 -->
<link 
  rel="preload" 
  href="noto-sans-sc-subset.woff2" 
  as="font" 
  type="font/woff2" 
  crossorigin
>
```

### 性能优化验证流程

1. **优化前基准测试**：使用 Lighthouse 记录基准分数和各项指标

2. **实施优化**：按照优化策略逐步实施

3. **优化后对比测试**：
   - 比较 Lighthouse 分数是否有显著提升
   - 检查 Core Web Vitals 是否达到良好标准

4. **上线监控**：
   - 监控是否出现性能回退
   - 持续追踪 Core Web Vitals（FCP、LCP、FID、CLS）
   - 及时发现并解决问题

## 常见误区

### ❌ 错误观点

1. **"TTI 是最重要的指标"**
   - ✅ TTI 已被弃用，应关注 LCP、FID、CLS

2. **"FMP 还在使用"**
   - ✅ FMP 已于 2020 年停止支持，应用 LCP 替代

3. **"HTTP 请求越少越好"**
   - ✅ HTTP/2 支持多路复用，小文件合并反而可能降低性能

4. **"所有资源都需要缓存"**
   - ✅ HTML 文件应设置较短的缓存时间或使用协商缓存

5. **"图片越清晰越好"**
   - ✅ 应根据设备分辨率和视口大小提供合适尺寸的图片
