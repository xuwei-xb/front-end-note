# 文档协同技术笔记

## 一、概述

### 1.1 什么是文档协同？

文档协同，是指 **多个用户** 在 **不同的设备** 对 **同一份文档** 进行实时编辑协作。

### 1.2 需要解决的问题

- **防止冲突**：多用户同时编辑时避免数据覆盖
- **数据一致性**：确保所有用户看到相同的文档状态
- **实时同步**：将修改即时同步给所有协作者

### 1.3 常见解决方案对比

| 方案 | 实时性 | 并发编辑 | 实现复杂度 | 人工干预 | 适用场景 |
|:---|:---:|:---:|:---:|:---:|:---|
| 读写锁 | 低 | ❌ | ⭐ 简单 | ❌ | 低频编辑、审批流程 |
| diff-patch | 中 | ⚠️ | ⭐⭐ 中等 | ⚠️ | 异步协作、版本管理 |
| OT | 高 | ✅ | ⭐⭐⭐ 复杂 | ❌ | 强实时、中心化系统 |
| CRDT | 高 | ✅ | ⭐⭐⭐ 复杂 | ❌ | 离线支持、分布式系统 |

## 二、方案一：读写锁

### 2.1 核心概念

通过 **写锁** 和 **读锁** 管理文档访问权限：

- **写锁**：独占编辑权，其他用户无法修改
- **读锁**：文档被编辑时，其他用户仍可只读访问

### 2.2 与操作系统读写锁的区别

| 类型 | 写锁时 | 读锁时 |
|:---|:---|:---|
| 操作系统 | 禁止读和写 | 允许并发读，禁止写 |
| 文档协同 | 允许只读 | 允许并发读，允许写锁排队 |

### 2.3 优缺点

**优点**：实现简单、完全避免冲突、读取并发

**缺点**：写锁期间用户体验差、可能饥饿现象

### 2.4 适用场景

- 实时性要求不高
- 内容稳定、变动频率低（90%查看，10%编辑）
- 强调强一致性（如审批流程文档）

---

## 三、方案二：diff-patch 合并

### 3.1 核心概念

通过计算文档差异来合并修改，类似 Git 合并：

1. **diff**：比较编辑内容与原始文档的差异
2. **patch**：生成格式化更新补丁
3. **冲突检测**：无法自动合并时手动解决

### 3.2 优缺点

**优点**：实现相对简单、支持并发编辑、版本可追溯、带宽小

**缺点**：冲突需人工处理、合并精度有限

### 3.3 适用场景

- 小范围协作编辑
- 文档结构稳定
- 异步协作与离线编辑
- 注重版本可追溯性

## 四、方案三：OT 算法

### 4.1 发展史

| 年份 | 里程碑 |
|:---:|:---|
| 1989 | OT 算法正式提出 |
| 2006 | Google Docs 首次商业应用 |
| 2011 | Office 365 基于 OT 实现协同 |
| 2012 | Quill 编辑器开源，Delta 数据模型 |
| 2013 | ShareDB 开源 |

### 4.2 核心思想

**OT = 操作（Operation）+ 转换（Transformation）**

每个用户的操作被记录并传播，通过**服务端转换**保证最终一致性。

### 4.3 操作描述（以 Quill Delta 为例）

```json
{
  "ops": [
    { "insert": "Gandalf", "attributes": { "bold": true } },
    { "insert": " the " },
    { "insert": "Grey", "attributes": { "color": "#cccccc" } }
  ]
}
```

| 操作 | 说明 | 示例 |
|:---|:---|:---|
| `insert` | 插入文本 | `{ "insert": "Hello" }` |
| `retain` | 保留字符 | `{ "retain": 7, "attributes": { "bold": null } }` |
| `delete` | 删除字符 | `{ "delete": 4 }` |

### 4.4 转换示例

```
初始文档: abc
用户A: 在位置1插入X
用户B: 在位置0插入Y

      abc
      ↓
  ┌───────────┐
  │    abc    │
  └───────────┘
      /    \
A:Insert(1,'X')  B:Insert(0,'Y')
     ↓              ↓
   aXbc           Yabc
     ↓              ↓
  OT变换          OT变换
     ↓              ↓
应用B'           应用A'
     ↓              ↓
  YaXbc          YaXbc
   ✅              ✅
```

### 4.5 OT 的局限性

- **强一致性设计**：所有操作需在服务端转换
- **依赖中央服务器**：不适合分布式系统
- **网络要求高**：异常可能导致转换失败

## 五、方案四：CRDT 算法

### 5.1 为什么需要 CRDT？

解决 OT 的局限性：去中心化、离线支持、网络容错。

### 5.2 CAP 理论

- **C（Consistency）**：一致性
- **A（Availability）**：可用性
- **P（Partition tolerance）**：分区容忍性

在分布式系统中，P 必须有，只能在 C 和 A 中选其一：
- **OT 选择 C**：强一致性
- **CRDT 选择 A**：最终一致性

### 5.3 核心思想

> 只要所有副本 **最终** 都收到相同的信息，不管谁先谁后，**最终状态一定一样**。

### 5.4 数据结构设计

每个字符带唯一 ID：

```js
{ char: 'x', id: [position_id, client_id] }
```

**示例**：初始文档 `abc`

| 字符 | ID |
|:---:|:---:|
| a | [1, A] |
| b | [2, A] |
| c | [3, A] |

用户A在a后插入X → ID `[1.5, A]`

用户B在开头插入Y → ID `[0.5, B]`

**合并**：按ID排序 → `YaXbc`

### 5.5 数据结构特点

| 特点 | 公式 | 作用 |
|:---|:---|:---|
| 可交换性 | `merge(a,b) === merge(b,a)` | 无视消息顺序 |
| 可结合性 | `merge(merge(a,b),c) === merge(a,merge(b,c))` | 不限制合并顺序 |
| 幂等性 | `merge(state, state) === state` | 支持消息重发 |

### 5.6 CRDT 类型

#### CvRDT（状态型）

同步**整个状态**，通过 `merge()` 函数合并。

**特点**：抗延迟/断网强、需完整状态

**代表**：G-Counter、OR-Set、Graph CRDT

#### CmRDT（操作型）

同步**操作**，操作可交换，顺序无关。

**特点**：操作必须可靠送达、无需完整状态

**代表**：RGA、Logoot、Yjs

| 对比 | CvRDT | CmRDT |
|:---|:---|:---|
| 同步内容 | 状态 | 操作 |
| 合并方式 | merge()函数 | 顺序无关执行 |
| 网络要求 | 支持离线 | 操作需可靠送达 |
| 实现复杂度 | 简单 | 复杂 |

### 5.7 经典 CRDT 数据结构

| 名称 | 类型 | 特点 | 支持操作 |
|:---|:---|:---|:---|
| G-Counter | CvRDT | 只能递增，每个副本维护自己计数 | 增加 |
| PN-Counter | CvRDT | 支持加减，用两个G-Counter | 增/减 |
| G-Set | CvRDT | 只能添加，不能删除 | 添加 |
| 2P-Set | CvRDT | 添加+删除（删除不可恢复） | 增/删 |
| LWW-Element-Set | CvRDT | 带时间戳，最后写入者生效 | 增/删 |
| OR-Set | CvRDT | 唯一ID跟踪添加/删除 | 增/删 |

**G-Counter 示例**：

```js
// 副本A: { A: 4 }
// 副本B: { B: 5 }
// 合并后: { A: 4, B: 5 }
// 总值: 9
```

**应用场景**：点赞数、浏览量、分布式日志计数

### 5.8 CRDT 实现对比

| 算法 | ID 结构 | 特点 |
|:---|:---|:---|
| RGA | [前一字符ID, 副本ID] | 链表式插入 |
| Logoot | [路径ID, 用户ID] | ID可排序且稀疏 |
| Yjs | 类似Logoot + 优化 | 偏移机制 |

## 六、OT vs CRDT 对比

| 对比维度 | OT | CRDT |
|:---|:---|:---|
| **一致性** | 强一致性 | 最终一致性 |
| **架构** | 需要中央服务器 | 去中心化 |
| **网络要求** | 高 | 低 |
| **离线支持** | ❌ | ✅ |
| **冲突处理** | 服务端转换 | 客户端自动合并 |
| **复杂度分布** | 服务端重、客户端轻 | 服务端轻、客户端重 |
| **数据模型** | 需实现不同转换算法 | 只需转化数据结构 |
| **代表实现** | Google Docs, ShareDB | Yjs, Automerge |

**选择建议**：
- 强一致性、高实时性 → OT
- 离线支持、去中心化 → CRDT

## 七、实践：Yjs 框架

### 7.1 架构设计

```
┌─────────────────────────────────────────────┐
│  编辑器适配层                                │
│  ProseMirror / Tiptap / Slate / Monaco      │
├─────────────────────────────────────────────┤
│  Yjs 核心                                   │
│  Y.Text / Y.Map / Y.Array / Y.Xml           │
├─────────────────────────────────────────────┤
│  通信与存储                                  │
│  y-websocket / y-webrtc / y-indexeddb       │
└─────────────────────────────────────────────┘
```

### 7.2 编辑器绑定

| 编辑器 | 绑定模块 |
|:---|:---|
| ProseMirror / Tiptap | `y-prosemirror` |
| Quill | `y-quill` |
| Monaco | `y-monaco` |
| CodeMirror | `y-codemirror.next` |

### 7.3 基本使用

```js
import * as Y from "yjs";

// 创建文档副本
const doc = new Y.Doc();

// 创建共享文本
const text = doc.getText("myText");
text.insert(0, "Hello Yjs");

console.log(text.toString()); // "Hello Yjs"
```

**多副本同步**：

```js
const doc1 = new Y.Doc();
const doc2 = new Y.Doc();

const text1 = doc1.getText("text");
const text2 = doc2.getText("text");

text1.insert(0, "Hello");

// 同步到 doc2
const update = Y.encodeStateAsUpdate(doc1);
Y.applyUpdate(doc2, update);

console.log(text2.toString()); // "Hello"
```

### 7.4 结构化数据

#### Y.Map

```js
const profile = doc.getMap("profile");
profile.set("name", "Alice");
profile.set("age", 30);

console.log(profile.toJSON()); // { name: "Alice", age: 30 }
```

#### Y.Array

```js
const list = doc.getArray("list");
list.push(["hello", "world"]);

console.log(list.toArray()); // ["hello", "world"]
```

#### 嵌套结构

```js
const users = doc.getMap("users");

const user1 = new Y.Map();
const tags = new Y.Array();
tags.push(["admin", "editor"]);
user1.set("tags", tags);

users.set("user1", user1);

console.log(users.toJSON());
// { user1: { tags: ["admin", "editor"] } }
```

**获取内容方式**：

| 类型 | 方法 |
|:---|:---|
| Y.Text | `text.toString()` |
| Y.Map | `map.toJSON()` |
| Y.Array | `array.toArray()` |

### 7.5 监听机制

| 方法 | 监听范围 | 用途 |
|:---|:---|:---|
| `observe(cb)` | 当前结构 | 精准字段监听 |
| `observeDeep(cb)` | 当前+所有子结构 | 嵌套结构统一处理 |

**observe 示例**：

```js
text.observe((event) => {
  event.changes.delta.forEach((change) => {
    if (change.insert) console.log("插入:", change.insert);
    if (change.delete) console.log("删除:", change.delete, "个字符");
  });
});
```

**observeDeep 示例**：

```js
root.observeDeep((events) => {
  events.forEach((e) => {
    console.log("变化位置:", e.path.join(" > "));
    
    if (e.target instanceof Y.Text) {
      // 处理文本变化
    }
    if (e.target instanceof Y.Array) {
      // 处理数组变化
    }
  });
});
```

**path 说明**：

```
rootMap (Y.Map)
  ├── title (Y.Text)          → path: ["title"]
  └── content (Y.Map)
        └── author (Y.Text)   → path: ["content", "author"]
```

### 7.6 标签页同步（BroadcastChannel）

```js
const channel = new BroadcastChannel("my-room");

// 发送更新
doc.on("update", (update) => {
  channel.postMessage(update);
});

// 接收更新
channel.onmessage = (event) => {
  Y.applyUpdate(doc, event.data);
};
```

### 7.7 核心优势

1. **冲突处理稳健**：基于CRDT，无需中央协调
2. **协作者状态同步**：内置Awareness模块
3. **离线编辑支持**：配合y-indexeddb持久化
4. **版本快照**：支持回溯
5. **高并发**：支持数十到上百人实时编辑

## 九、案例：AFFiNE 为何选择 CRDT

AFFiNE 是一个开源协同编辑器（Notion + Miro + Obsidian 混合风格），选择 CRDT 的原因：

### 9.1 去中心化

无需中心节点确定操作顺序，支持 **local-first** 架构，甚至端到端加密。

### 9.2 性能表现

经优化后吞吐量已超越传统OT实现，延迟容忍度更高。

### 9.3 灵活性

Yjs 支持更广泛的数据类型（Text、Array、Map、树形结构），接入成本更低。

**结论**：CRDT，特别是 Yjs，是构建协同编辑器的值得考虑的选项。
